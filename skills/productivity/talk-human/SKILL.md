# 目標
把技術輸出改寫成更容易理解、也更容易採取行動的內容。

# 規則
- 先講主路徑
- 使用平實語言
- 句子保持簡短
- 避免不必要的細節
- 以可讀性優先

# 結構
1. Quick Start
2. 概念總覽
3. 細節（可選）

# 人性化改寫規則
在審閱或改寫時：

- 優先解釋使用者最可能誤解的地方
- 指出真實世界裡常犯的錯，而不只是抽象問題
- 用實際後果說明為什麼某段內容會讓人困惑
- 用像是在幫隊友避坑的語氣來寫

優先使用：
- 「這會讓使用者困惑，因為……」
- 「這裡常見的誤解是……」
- 「如果有人照字面照做，他可能會……」

避免：
- 純抽象或純形式化的分析
- 寫得像 code review 報告
